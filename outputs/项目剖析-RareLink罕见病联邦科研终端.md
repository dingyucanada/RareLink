# RareLink 稀联：罕见病联邦科研智能体终端项目剖析

## 1. 结论

这个方向具有很高的获奖上限，技术深度、行业价值和 DGX Spark 平台适配性都明显强于普通内容创作或视频分析项目，适合冲击技术创新类奖项。

但原始表述需要调整：

- 不建议称为“罕见病诊断终端”；
- 不建议在比赛阶段宣称可直接服务临床决策；
- 建议定位为“罕见病多中心科研与本地研究分析终端”；
- 不以“部署 NVIDIA FLARE”为创新点，而以“Agent 自动创建、执行、评审和复现多中心研究”为创新点。

推荐题目：

> **RareLink 稀联：每个科室一台 DGX Spark 的罕见病联邦科研智能体终端**

一句话介绍：

> 每个医院科室部署一台 DGX Spark，患者影像和研究数据始终留在本地；NVIDIA FLARE 只交换模型更新和受控统计结果，Step 3.7 驱动的科研 Agent Team 自动完成研究协议设计、跨院数据可行性分析、联邦训练、方法比较、隐私审计和研究报告生成。

核心口号：

> **数据不出科室，模型跨院学习，Agent 自动完成一项可复现的多中心研究。**

## 2. 为什么方向成立

### 2.1 罕见病天然需要多中心协作

单个医院的罕见病病例通常有限，难以覆盖患者差异、成像设备、检查协议和疾病亚型。多中心联合研究能够扩大样本范围，但原始医疗数据受到隐私、伦理、授权和机构治理限制，难以直接集中。

联邦学习适合这个矛盾：训练任务被发送到各站点，模型在本地数据上更新，站点只返回受控的模型更新或统计结果，原始数据不集中。

需要避免夸大：联邦学习不会自动解决样本少、标注差、站点偏差或隐私问题；它只提供“不集中原始数据的协作计算机制”。

### 2.2 “一盒一科室”是合理产品形态

DGX Spark 不是高速数据中心 GPU，但具备 128GB CPU/GPU 统一内存，适合在科室本地同时运行：

- 三维医学影像预处理；
- MONAI 训练或微调任务；
- 本地推理和标注辅助；
- NVIDIA FLARE Client；
- 数据质控和研究 Agent；
- 实验跟踪、审计和本地检索。

其价值不是单项训练速度领先，而是将一个完整的研究节点压缩到桌面设备中，并把数据处理边界留在科室。

### 2.3 NVIDIA 技术栈高度匹配

NVIDIA FLARE 是开源、领域无关的联邦计算 SDK，支持模拟和真实部署、MONAI 集成、联邦统计、差分隐私、同态加密、PSI、站点策略和审计日志。官方已有三维医学影像、脑肿瘤分割、前列腺分割和医疗 VLM 联邦微调示例。[NVIDIA FLARE](https://nvidia.github.io/NVFlare/)、[安全能力](https://nvidia.github.io/NVFlare/security/)

MONAI 提供医学影像训练、标注、模型 Bundle 和部署基础；VISTA-3D 可作为三维医学影像分割与标注基础模型。[MONAI](https://github.com/Project-MONAI/MONAI)、[VISTA](https://github.com/Project-MONAI/VISTA)

## 3. 最大问题：单纯联邦学习并不新

如果项目只完成以下流程，创新性不足：

```text
公开数据集切成三份
→ 启动三个 NVIDIA FLARE Client
→ 使用 FedAvg 聚合
→ 展示 Dice 提升
```

原因：

- NVIDIA FLARE 已有 MONAI 和 BraTS 联邦学习示例；
- 差分隐私、FedAvg、FedProx、SCAFFOLD 等也已有教程；
- 2026 年 NVIDIA 已发布 FLARE Auto-FL，用 Agent 在固定预算和可比协议下搜索联邦策略；
- 只运行官方示例仍然属于部署集成项目。

NVIDIA FLARE Auto-FL 已支持受约束地尝试 FedAvg、FedOpt、FedAdam、SCAFFOLD、FedProx 和有限架构搜索，并使用固定 benchmark contract 与 experiment ledger 保证实验可比和可复现。[NVIDIA FLARE Auto-FL](https://developer.nvidia.com/blog/?p=118222)

因此必须在 Auto-FL 之上增加**罕见病多中心科研工作流**，而不是重复实现 Auto-FL。

## 4. 真正的创新：Agentic Federated Research

项目的核心不是“联邦训练一个模型”，而是：

> 用户提出一个研究问题后，Agent Team 自动创建一项受约束、可审计、可复现的多中心联邦研究。

完整闭环：

```text
研究问题
  → 自动生成研究协议草案
  → 各站点进行隐私保护的可行性统计
  → 发现数据缺失、标签差异和站点偏移
  → 建立固定实验合同与评估指标
  → 运行本地基线和联邦基线
  → Auto-FL 在受限范围内比较策略
  → 统计 Agent 审查站点表现、校准和不确定性
  → 隐私 Agent 审查输出与隐私预算
  → 生成研究报告、模型卡和实验账本
```

### 4.1 关键创新一：联邦可行性研究

正式训练前，各科室只返回经过规则控制的汇总信息：

- 符合纳入标准的病例数；
- 缺失模态比例；
- 扫描参数分布；
- 标签完整性；
- 年龄段或疾病亚型的分桶计数；
- 图像尺寸、强度和质量统计。

系统不先假定数据可以联合，而是由 Agent 判断：

- 样本量是否足以启动研究；
- 是否存在过小分组导致重识别风险；
- 各站点标签定义是否一致；
- 是否需要修改研究问题或合并分组；
- 哪些指标可以安全公开。

### 4.2 关键创新二：受约束的科研 Agent

Agent 不能自由修改实验，以避免通过改变测试集、预算或模型规模制造“虚假提升”。系统锁定：

- 数据划分；
- 训练轮数和总计算预算；
- 可用算法集合；
- 评价指标；
- 隐私预算；
- 模型参数规模上限；
- 允许修改的超参数范围。

所有实验记录到不可覆盖的实验账本：

```json
{
  "experiment_id": "FL-024",
  "hypothesis": "FedProx may reduce scanner-site drift",
  "change": {"strategy": "FedProx", "mu": 0.01},
  "fixed_budget": {"rounds": 5, "local_epochs": 1},
  "privacy": {"dp": true, "epsilon": 8.0},
  "metrics": {"mean_dice": 0.0, "worst_site_dice": 0.0},
  "decision": "pending_review"
}
```

### 4.3 关键创新三：从全局模型到科室个性化

罕见病多中心数据通常不是独立同分布。系统不只展示平均指标，还比较：

- 本地模型；
- FedAvg 全局模型；
- FedProx/SCAFFOLD 等异质性策略；
- 个性化联邦模型；
- 最差站点表现；
- 留一站点外部验证。

最终目标不是让平均分最高，而是避免某个小站点或特殊人群表现显著下降。

### 4.4 关键创新四：模型、数据和研究三重可追溯

最终生成：

- 研究协议与版本；
- 站点数据卡；
- 联邦实验账本；
- 模型卡；
- 隐私与安全报告；
- 每站点评价结果；
- 研究报告草案；
- 可复现实验配置。

## 5. 产品模式

### 5.1 科室本地研究分析模式

用于科研，不作为自动诊断：

- 导入去标识化 DICOM/NIfTI；
- 检查图像质量和元数据；
- 使用 MONAI 模型生成研究用分割建议；
- 由研究人员审核和修订标注；
- 查看本地模型和联邦模型的差异；
- 创建本地数据卡和研究队列。

### 5.2 多中心联邦研究模式

- 加入经过授权的研究项目；
- 审核研究协议和本地计算任务；
- 在本地运行训练或统计；
- 通过 NVIDIA FLARE 返回受控更新；
- 接收全局或个性化模型；
- 输出站点级评价和审计记录。

“分析和科研双用途”建议改写成“本地研究分析＋多中心联邦研究”，避免被理解为未经验证的临床诊断系统。

## 6. Agent Team

| Agent | 运行位置 | 职责 | 可见数据 |
|---|---|---|---|
| 研究主任 Agent（Step 3.7） | 协调端 | 将研究问题转成研究协议和任务图 | 公开资料、去标识化汇总信息 |
| 站点数据管家 Agent | 每个 Spark | 检查格式、缺失、异常值、授权和纳入条件 | 本地原始数据 |
| 表型/影像质控 Agent | 每个 Spark | 生成本地数据卡和质量报告 | 本地影像和元数据 |
| 联邦实验 Agent | 协调端 | 建立基线并调用受约束 Auto-FL | 模型更新和实验指标 |
| 统计审稿 Agent | 协调端 | 比较均值、最差站点、校准和置信区间 | 汇总指标 |
| 隐私红队 Agent | 各端 | 检查输出字段、过小分组、隐私预算和异常更新 | 本站输出及协议 |
| 科研写作 Agent | 协调端 | 根据实验账本生成报告和模型卡 | 已批准结果 |
| 人类 PI/站点管理员 | 各端 | 批准协议、任务、数据使用和最终结论 | 授权范围内数据 |

Step 3.7 不应读取原始患者影像、病历或可识别信息。若使用比赛 API Plan，只向其提供公开研究问题、Schema 和经过批准的聚合指标。

## 8. 隐私与安全边界

### 8.1 联邦学习不等于自动匿名

虽然原始数据不离开站点，但模型更新仍可能泄露信息，尤其在样本极少、单站点病例很少或攻击者能观察单独更新时。因此不能宣传“绝对隐私”。

需要实现或明确说明：

- PKI 双向认证；
- 角色与站点授权；
- TLS 通信；
- 本地输出白名单；
- 最小参与站点数；
- 小样本计数抑制；
- 梯度裁剪；
- 差分隐私及隐私预算；
- 安全聚合或同态加密的可选方案；
- 全程审计日志；
- 异常更新和投毒检测；
- 删除、撤回和研究终止流程。

NVIDIA FLARE 已提供 PKI、角色、联邦授权、站点策略、过滤器、审计、差分隐私、同态加密、PSI 和机密计算接口，但项目仍需根据研究协议正确配置。[FLARE Security](https://nvidia.github.io/NVFlare/security/)

### 8.2 中国医疗数据合规提示

医疗健康信息属于敏感个人信息，处理需要具备特定目的、充分必要性和严格保护措施；实际医院落地还涉及伦理审查、知情同意或相应合法基础、数据分类分级、访问控制、留痕和跨境规则。[《个人信息保护法》第二十八至三十条](https://www.npc.gov.cn/WZWSREL25wYy9jMi9jMzA4MzQvMjAyMTA4L3QyMDIxMDgyMF8zMTMwODguaHRtbD9yZWY9aW1i)、[国家健康医疗大数据管理办法](https://www.nhc.gov.cn/mohwsbwstjxxzx/s8553/201809/742308fad8ed47fb85b9675ee4b6c6eb.shtml)、[医疗卫生机构网络安全管理办法](https://www.nhc.gov.cn/guihuaxxs/c100133/202208/8a23d01133214a779879094dd20cd383.shtml)

比赛 Demo 只使用公开、去标识化、明确允许研究使用的数据，不接触真实医院患者数据。

## 9. 推荐疾病与数据

不建议泛化为所有罕见病。MVP 应聚焦一个明确任务：

> **儿童高级别胶质瘤多中心 MRI 分割研究。**

理由：

- 符合罕见病/罕见肿瘤的小样本多中心痛点；
- 输入为多序列三维 MRI，能体现本地算力；
- 分割结果可三维可视化，Demo 比表格分类更直观；
- MONAI 和 NVIDIA FLARE 已有脑肿瘤分割技术基础；
- 可以使用公开去标识化数据。

可用数据：

- [BraTS-PEDs](https://www.cancerimagingarchive.net/collection/brats-peds/)：多机构儿童脑肿瘤 MRI，包含四种常规 MRI 序列及专家修订标注，需遵守 TCIA/Synapse 使用政策和引用要求；
- [DFCI-BCH-BWH-PEDs-HGG](https://www.cancerimagingarchive.net/collection/dfci-bch-bwh-peds-hgg/)：61 例儿童高级别胶质瘤多序列 MRI，适合数据探索，但 TCIA 页面说明对应分割标签另行发布，因此不应直接假定其具备可用分割标签；
- NVIDIA FLARE 官方 BraTS18 差分隐私示例可用作工程基线，但不能将成人 BraTS18 结果包装成儿童罕见病真实验证。

若 BraTS-PEDs 下载审批或预处理来不及，可用官方 BraTS18 示例完成技术验证，并明确标注为“联邦系统工程验证数据”，同时将儿童肿瘤作为目标应用而非已验证结论。

## 10. 科学评价设计

### 10.1 必须比较的基线

| 模型 | 作用 |
|---|---|
| Local-only | 各站点仅用本地数据训练，作为下界 |
| FedAvg | 最基本联邦基线 |
| FedProx 或 SCAFFOLD | 处理站点异质性 |
| Personalized FL | 评估个性化是否改善特殊站点 |
| Centralized | 只在公开模拟数据中作为研究上界，不用于真实部署 |

### 10.2 评价指标

分割任务至少包含：

- Dice；
- HD95；
- Sensitivity；
- Specificity；
- 每站点指标；
- 最差站点指标；
- 均值和置信区间；
- 模型校准或不确定性；
- 通信量；
- 训练时间；
- 隐私预算和性能损失。

罕见病项目不能只展示平均 Dice。最差站点、罕见亚组和外部站点表现更重要。

### 10.3 需要模拟的数据异质性

- 不同站点样本量；
- 不同疾病亚型比例；
- 不同扫描设备或强度分布；
- 某站点缺少一种 MRI 模态；
- 标签边界标准差异；
- 小站点掉线或延迟。

所有模拟条件必须记录，不能将人为切分的数据宣称为真实医院分布。

## 11. Demo 设计

### 11.1 四分钟演示

| 时间 | 演示内容 |
|---|---|
| 0:00–0:25 | 地图显示三个医院科室节点，每个节点拥有少量不同分布病例 |
| 0:25–0:55 | 输入研究问题，Step 3.7 生成研究协议和固定实验合同 |
| 0:55–1:25 | 站点 Agent 返回受控统计；系统发现某站点缺少模态且病例过少 |
| 1:25–1:55 | 人类批准调整后的纳入标准和联邦实验计划 |
| 1:55–2:25 | 现场运行一个缩短的联邦轮次，数据包检查器证明没有传输 DICOM/NIfTI |
| 2:25–3:00 | 展示已完成实验账本：Local、FedAvg、FedProx/个性化策略对比 |
| 3:00–3:30 | 三维查看同一病例的本地与联邦模型分割差异及不确定区域 |
| 3:30–4:00 | 自动生成研究报告、模型卡、站点数据卡和隐私报告 |

### 11.2 三个高光镜头

1. 点击网络中的更新包，看到只有模型更新和批准字段，没有患者影像；
2. Agent 发现平均性能提升但最小站点下降，拒绝直接发布全局模型；
3. 从一句研究问题自动生成一套带实验账本和模型卡的可复现多中心研究包。

完整三维训练不适合现场等待。应提前运行完整实验，演示时明确展示“历史完成实验”；现场只运行一个小数据、短轮次验证通信和编排，不能将回放伪装成实时训练。

## 12. 十天 MVP

### 12.1 必须做

- 一个罕见肿瘤分割任务；
- 三个模拟医院 Client；
- 一个真实 NVIDIA FLARE Server；
- Local-only 与 FedAvg 基线；
- 一个异质性策略，如 FedProx；
- 联邦统计和数据质量检查；
- Step 3.7 研究协议 Agent；
- 受约束实验账本；
- 三维分割展示；
- 自动生成模型卡和研究报告；
- 明确的安全和合规边界。

### 12.2 可选增强

- 差分隐私；
- 个性化联邦学习；
- FLARE Auto-FL 策略搜索；
- 异常更新检测；
- 第二台真实设备作为远程 Client；
- MONAI Label 人工审核。

### 12.3 不做

- 不接入真实医院系统；
- 不使用真实未授权患者数据；
- 不宣传临床诊断能力；
- 不覆盖所有罕见病；
- 不做基因、影像、病历全模态统一模型；
- 不承诺绝对隐私；
- 不在十天内完成生产级跨院部署。

## 15. 最容易失分的地方

1. 把一个公开数据集随机切成三份，却宣称是真实三医院研究；
2. 只运行 NVIDIA FLARE 官方示例；
3. 用“数据不出院”代替完整隐私分析；
4. 让 Step 3.7 接触原始病历或影像；
5. 宣称模型可诊断罕见病；
6. 只报告平均准确率，不报告每站点和最差站点表现；
7. Auto-FL 可以任意修改测试集、预算或模型规模；
8. 现场等待长时间训练，没有可看的 Agent 决策过程；
9. 三个 Client 全在同一台机器，却不说明是模拟部署；
10. 没有人工批准、研究终止和审计机制。

## 16. 最终建议

这个题目值得作为主赛题，但必须采用以下精确定位：

> **RareLink 不是罕见病自动诊断系统，也不是 NVIDIA FLARE 部署演示；它是一套以 DGX Spark 为科室本地节点、以 NVIDIA FLARE 为隐私协作底座、以 Step 3.7 为科研总导演的多中心研究操作系统。**

最终交付重点不是一个更高的 Dice，而是证明 Agent 能在不汇聚原始医疗数据的前提下，创建、运行、评审并打包一项可复现的多中心罕见病研究。
