# DGX Spark Hackathon 赛事需求

## 1. 赛事概述

- **赛事主题：**让 Agent 创作一切
- **技术方向：**多模态 Agent、Agent Team、本地大模型部署与优化
- **核心平台：**NVIDIA DGX Spark
- **核心模型：**Step 3.7（StepFun 阶跃星辰）
- **项目目标：**基于 DGX Spark 和 Step 3.7 构建本地 Agent Team，使其具备感知、理解、自主创作、协同执行和迭代优化能力，并在内容生产、自动化工作流或多模态创意场景中完成实际落地。

项目不应停留在普通问答或固定流程调用。完整方案应体现：任务理解、自动规划、多 Agent 分工、工具调用、内容生成、质量检查、反馈修改和最终交付。

## 2. 核心要求

### 2.1 Agent 能力

项目应根据实际场景实现以下能力：

- 接收并理解文本、图片、音频、视频或文件等多模态输入；
- 将用户目标拆解为可执行任务；
- 调度多个专业 Agent 分工或并行执行；
- 调用本地模型、代码、知识库及业务工具；
- 检查中间结果并处理失败、冲突和低质量输出；
- 根据评审结果自动修改，形成创作与优化闭环；
- 输出可直接使用的内容、设计、代码、视频、报告或工作流成果。

推荐的最小协作闭环：

```text
用户目标
  → 多模态理解
  → 任务规划
  → Agent 分工与执行
  → 工具和模型调用
  → 结果评审
  → 定向返工与优化
  → 最终成果
```

### 2.2 本地部署

项目应明确说明：

- 哪些模型和服务部署在 DGX Spark；
- Step 3.7 在规划、推理、创作、评审或调度中的具体作用；
- Agent Team 如何访问本地模型和工具；
- GPU、统一内存、并行计算和本地推理能力如何被利用；
- 本地部署在隐私、延迟、成本、可控性或离线运行方面的价值；
- 外部云端服务的使用范围及其不可替代性。

DGX Spark 和 Step 3.7 应承担核心功能，不能只在文档中出现或作为非关键附加组件。

### 2.3 模型与系统优化

优化方案需说明实施位置、技术原理和实际效果。可选方向包括：

- FP16、BF16、FP8、INT8 或 INT4 量化；
- TensorRT-LLM、NVIDIA NIM 或 Triton 推理服务；
- KV Cache、提示词缓存和上下文裁剪；
- 批处理、并发调度、模型常驻和资源复用；
- Agent 并行执行、超时控制、失败重试和任务状态恢复；
- 大小模型路由、检索结果压缩和无效循环控制；
- 面向任务的提示词、工具调用、微调或偏好优化。

优化结果应尽量提供量化对比，包括：

- 首字延迟；
- 每秒生成 Token 数；
- 单次任务总耗时；
- 峰值 GPU、显存或统一内存占用；
- 并发任务数；
- 任务成功率；
- Agent 平均迭代次数；
- 优化前后的输出质量。

## 7. DGX Spark 使用与故障排查（内部信息，公开前删除）

> 本节包含队伍服务器信息，不得提交到公开仓库、征文、视频或公开截图。登录密码不在本文记录；已公开或发送过的密码应立即更换。

### 7.1 队伍机器

| 项目 | 信息 |
|---|---|
| 队名 | 发现新的大陆 |
| 主机名 | `spark-76` |
| 用户名 | `Developer` |
| 公网 IP | `106.13.186.155` |
| SSH 公网端口 | `6076` |

标准 OpenSSH 连接命令：

```bash
ssh -p 6076 Developer@106.13.186.155
```

Windows、macOS 和 Linux 的 OpenSSH 均使用 `-p` 指定端口。`ssh 用户名@IP:端口` 通常不是有效的 OpenSSH 命令格式。

### 7.2 端口映射

| 内网地址 | 公网地址 |
|---|---|
| `192.168.110.76:22` | `106.13.186.155:6076` |
| `192.168.110.76:7000` | `106.13.186.155:7076` |
| `192.168.110.76:8888` | `106.13.186.155:8076` |
| `192.168.110.76:9000` | `106.13.186.155:9076` |

服务必须按需要监听 `0.0.0.0`，不能只监听 `127.0.0.1`。例如，内网服务监听 `0.0.0.0:8888` 时，其映射地址为：

```text
http://106.13.186.155:8076
```

实际可访问性取决于服务状态、防火墙和平台端口策略。

### 7.3 无法连接时的排查顺序

1. 检查公网 IP、SSH 端口和用户名；
2. 检查机器是否在线；
3. 检查内存占用，连接失败常由内存溢出导致；
4. 检查是否存在残留模型或异常进程；
5. 检查服务是否监听正确地址和端口；
6. 检查公网端口是否与内网端口正确对应；
7. 查看服务日志及 GPU、内存状态。

大文件应使用组委会允许的共享存储、对象存储或模型仓库，不要通过 SSH 直接上传。
