## 2. 移植目标、范围与架构推理

### 2.1 为什么要移植到 DGX Spark

RareLink 的核心工作负载由四部分组成：三维四模态 MRI 的局部训练、多个科室/站点训练任务的调度、联邦模型协作、安全审计与科研 Agent 编排。它们的共同特征是数据敏感、显存/内存占用大、任务间需要共享本地上下文，但并不依赖高带宽把影像持续传到云端。

因此，DGX Spark 的价值不被表述为“传输视频或影像更快”，而是：

1. **本地计算靠近数据。** 每个未来医院科室可将 Spark 视为一个本地科研终端，医学影像和标签留在院内文件系统与训练进程中。
2. **统一内存适合多组件并存。** 3D MONAI 模型、FLARE 客户端、API、数据库和 Agent 控制任务可共处一台节点；比赛原型通过串行化重型训练避免统一内存争抢。
3. **CUDA 原生生态缩短工程路径。** PyTorch、MONAI、NVIDIA FLARE、Opacus 能在同一 ARM64/CUDA 环境中形成真实链路，避免把“模拟联邦”包装为真实训练。
4. **边缘/科室部署形态可扩展。** 比赛中一台 Spark 承载三个逻辑站点；真实部署时，每个站点应替换为一台在医院网络与身份体系内独立运行的 Spark Client。

### 2.2 本次部署范围

本次已验证的是一台 Spark 上的三逻辑站点联邦训练，以及 Spark 与一台 Mac 之间的 FLARE mTLS 通信。没有接入真实医院、PACS、FHIR、DICOMweb、生产 SSO 或真实临床数据。

```text
浏览器研究控制台
        │
React / Vite :8888 ──反向代理── FastAPI :9000
                                      │
                    审计账本 / 研究状态机 / 策略网关
                         │              │
               Step 3.7（仅脱敏文本）   │
                                        ▼
                     NVIDIA FLARE Server + Site A/B/C
                                        │
                     MONAI SegResNet + PyTorch CUDA
                                        │
                    合成四模态 NIfTI / 本地模型产物
```

推理原则是：**模型训练路径与 Agent 文本路径必须隔离。** 原始影像、标签、患者标识、DICOM UID、文件路径和站点小样本明细不进入 Step 3.7；对外生成的研究报告也必须经输出安全门校验。

### 3.2 部署与运行方式

部署采用 Spark 节点内的容器化运行：后端容器使用 host network 暴露 API，前端容器/宿主 Vite 服务通过 API 代理访问后端；赛事分配端口仅用于浏览器展示与 SSH 运维。源码以小体积形式同步，医学影像、模型权重、数据库、训练产物均在 Spark 内生成和保存，没有通过 SSH 传输大文件。

移植中遵循以下工程决策：

| 风险 | 处理 | 推理 |
| --- | --- | --- |
| 三个 3D 模型同时训练导致统一内存压力 | 同机逻辑站点串行训练，`num_threads=1` | 比赛目标是证明联邦闭环；在共享节点抢占并发会降低稳定性并影响他人 |
| Docker daemon 本机代理不可用，无法拉取新镜像 | 不改共享节点 daemon，复用已存在 ARM64 CUDA 运行环境 | 节点治理优先于方便；保留 `Dockerfile.spark` 供网络恢复后按官方镜像重建 |
| Docker 用户组权限不足 | 使用受控的 `sudo docker`，不修改用户组/daemon | 不改变共享赛事节点的全局安全配置 |
| Step API 密钥风险 | Spark 不保存明文密钥；未配置时使用确定性模板 Agent | 联邦训练不应因远程 LLM 服务不可用而失效 |
| PyTorch 对 GB10 12.1 的提示 | 将 warning 记录为限制，基于实际 kernel/训练成功继续验证 | “能跑”不等于“官方完整支持”；后续应换 NVIDIA 推荐 NGC 镜像复测 |

### 3.3 服务可用性结果

Spark 内网服务使用前端端口 8888、后端端口 9000。验证中 API health 返回 `{"status":"ok","service":"rarelink"}`，Swagger 文档可访问，React 首页返回 HTTP 200，前端 `/api` 代理正常工作。API、数据库状态机、后台训练子进程、训练日志、指标回写和全局模型产物均已打通。

## 4. 端到端训练与控制平面实验

### 4.1 合成数据与模型链路

为避免用患者数据进行比赛开发，系统在 Spark 本地生成三个逻辑站点的合成病例。每例包含 T1、T1CE、T2、FLAIR 四个 32×32×32 NIfTI 模态和分割标签，并写入 SHA-256 清单。模型为 MONAI 3D SegResNet，使用 DiceCE 损失、Dice 与 HD95 指标。

单站点 GPU 冒烟配置为 Site A、4 例数据（3 训练、1 验证）、1 epoch。结果为：epoch loss `1.844093`，mean foreground Dice `0.037142`，HD95 `14.123105`，并产生模型 `.pt` 产物。

这些数值很低，原因是样本极少、体素尺寸小、仅训练一个 epoch、且合成数据不是临床分布。报告保留低数值而不做美化，因为该实验要回答的是“GPU、数据转换、损失、评估、模型持久化是否真实贯通”，不是“模型是否具有医学诊断能力”。

### 4.2 联邦与控制平面闭环

系统工作流为：

```text
DRAFT → PROTOCOL_REVIEW → FEASIBILITY_RUNNING → FEASIBILITY_REVIEW
→ CONTRACT_LOCKED → TRAINING_RUNNING → RESULTS_REVIEW
```

在正式演练中，研究由 FastAPI 创建，Agent 生成受控协议，人工批准并锁定训练合同，后台任务进入统一内存队列，NVIDIA FLARE 依次完成三个逻辑站点的本地训练与聚合，指标、日志、全局模型与审计事件回写数据库。

早期单轮实测为：

| 策略 | Mean Dice | Worst-site Dice | Mean HD95 | 结论 |
| --- | ---: | ---: | ---: | --- |
| FedAvg，1 轮 | 0.074098 | 0.064549 | 18.097708 | 三个站点均提交更新并生成 `FL_global_model.pt` |
| FedProx，1 轮，`mu=0.01` | 0.026749 | 0.023887 | 15.575719 | 真实 FedProx 路径、近端项与全局模型均已验证 |
| 控制平面 FedAvg 任务 | 0.104327 | 0.087986 | 14.908172 | 状态为 `COMPLETED`、进度 100%、研究进入 `RESULTS_REVIEW` |

**推理：** 单次、单轮、随机初始化的 FedAvg/FedProx 数值不具有策略优劣的统计意义。因此项目随后升级为对齐计算预算的五种子、多轮矩阵，而没有选取最好的一次单轮结果对外宣传。

## 5. 五种子、多轮稳定性实验

### 5.1 实验设计

配置固定为随机种子 `2026, 2027, 2028, 2029, 2030`；三个逻辑站点；联邦策略每轮每站点 1 个本地 epoch，共 3 轮；Local 基线自动执行 3 个 epoch，使每个策略获得相同的本地训练机会。策略包括 Local、FedAvg、FedProx、FedAvg+SVT 与 FedAvg+DP-SGD，共 `5 × 5 = 25` 个完整策略—种子组合。

这种设计解决两个常见误导：一是联邦策略训练次数比 Local 多；二是用单个随机种子掩盖训练波动。评估同时报告平均 Dice、最弱站点 Dice、HD95、最弱站点胜率、运行时间与峰值 GPU 内存，避免只取单一平均分数。

### 5.2 Spark 实测结果

| 策略 | Mean Dice（均值 ± 标准差） | Worst-site Dice（均值 ± 标准差） | HD95 均值 | 最弱站点胜率 | 平均运行时间 | 峰值 GPU 内存 |
| --- | ---: | ---: | ---: | ---: | ---: | ---: |
| Local | 0.049025 ± 0.013654 | 0.023893 ± 0.011558 | 14.514729 | 20% | 1.52 s | 24.38 MB |
| FedAvg | 0.080087 ± 0.080502 | 0.072276 ± 0.075325 | 14.390605 | 40% | 73.05 s | 20.73 MB |
| FedProx | 0.073442 ± 0.035380 | 0.051055 ± 0.030026 | 13.879725 | 20% | 72.68 s | 21.59 MB |
| FedAvg + SVT | 0.000000 ± 0.000000 | 0.000000 ± 0.000000 | 0.000000 | 0% | 73.21 s | 20.73 MB |
| FedAvg + DP-SGD | 0.041315 ± 0.052110 | 0.037724 ± 0.048620 | 18.970487 | 20% | 73.59 s | 55.78 MB |

相对 Local，FedAvg、FedProx、DP-SGD、SVT 分别在 `3/5`、`4/5`、`2/5`、`0/5` 个种子上改善最弱站点 Dice。

### 5.3 结果推理与结论

1. **当前演示候选为 FedAvg。** 它的平均最弱站点 Dice 为 `0.072276`，最弱站点胜率 `40%`，在本矩阵中处于最高。选择理由是公平性指标（最弱站点）而非只看均值。
2. **FedProx 值得保留为异质性候选。** 它在 `4/5` 种子上改善最弱站点，但平均胜率和均值不超过 FedAvg，说明小样本三轮设置下仍不能写成“FedProx 更好”。
3. **SVT 配置失败是有价值的负结果。** 在每次仅共享 1% 更新、`epsilon=0.1`、`noise_var=0.1` 的严格过滤参数下，五次实验均为 0。该结果说明隐私/过滤强度会损害效用，应作为参数选择的证据而不是删除失败数据。
4. **DP-SGD 带来可观的工程开销与效用压力。** 峰值 GPU 内存从约 20–24 MB 上升到 `55.78 MB`，Mean Dice 低于 FedAvg。这符合逐样本梯度计算的预期，不能将其包装为“零成本隐私”。
5. **不能做医学统计推断。** 五个种子、三轮和合成数据只能说明工程行为与配置敏感性；FedAvg 很宽的波动也提示需要在正式授权的公开数据上增加病例、轮数与重复次数。

## 6. 样本级 DP-SGD 与隐私会计

### 6.1 实现

`fedavg_dpsgd` 使用 Opacus 1.6.0 的 expanded-weights 模式，适配 MONAI SegResNet 的残差操作。每个站点本地训练中执行：逐样本梯度、`max_grad_norm=1.0` 裁剪、高斯噪声（`noise_multiplier=1.2`）、Poisson 采样；服务端再根据采样率、计划优化步数和联邦轮次使用 RDP Accountant 组成累计预算。

三轮配置下，每个站点的采样率为 `1/3`，保守计入 9 个计划优化步，在 `delta=1e-5` 时累计 `epsilon=6.076881`。

### 6.2 空批次边界与修复推理

Poisson 采样可能返回空批次。若把空批次送入 Conv3d，会报错；若简单跳过且不计费，则可能低估隐私预算。实现采取“空批次不更新模型，但按计划调用保守计入会计”的规则。另一个边界是首批为空时，Opacus 对 MONAI 字典样本的 dtype 推导异常；系统在 Poisson 抽样前使用一个已整理的本地样本预热 collator，随后在原失败种子上复验通过。

**准确结论：** 已为每家站点的**本地训练步骤**完成可复算的样本级 DP-SGD 会计。该结论不覆盖网络传输、联邦聚合、用户级/医院级邻接关系、端到端系统、成员推断攻击防护或临床隐私合规。当前 `secure_rng=false` 也已明确披露，故该实现是比赛工程证据而非生产级 DP 认证。

## 7. 两物理设备 mTLS 联邦安全演练

### 7.1 演练设计

NVIDIA FLARE Provision 工具在 Spark 生成 Root CA、Server 证书、Site A/B/C 各自证书/私钥、启动包签名及组织标识。首先完成 Spark 内 Server + 三 Client 的安全注册；然后将 Server 保留在 Spark，将 Site C 实际运行在一台 Mac 上，通过加密 SSH 隧道承载 FLARE mTLS 控制流。

测试包含三个必要条件：

| 场景 | 结果 | 所证明的内容 |
| --- | --- | --- |
| Mac Site C 首次注册 | 成功 | 跨物理设备连接与证书链可用 |
| 主动停止后再次注册 | 成功 | 客户端掉线后的重新加入路径可用 |
| 将 Site B 凭据用于声明为 Site C 的启动包 | 以非零状态被拒绝 | 启动包签名与身份一致性校验在负面对照中生效 |

报告只保留平台信息、哈希化设备指纹和通过/拒绝布尔值；不保存运行令牌、会话 ID、私钥、原始日志或认证材料。临时证书包和本机临时文件在证据固化后删除，Git 仓库不包含任何证书私钥。

### 7.2 推理与结论

与“同一台机器启动三个客户端”相比，这一演练增加了物理设备差异、跨设备连接、掉线重连和错误身份拒绝三个关键反例。因此它是更强的安全通信工程证据。

但结论必须限定为：**Spark–Mac 两物理设备的 mTLS 注册、重连与错误身份拒绝已完成。** 演练没有覆盖医院 WAN 抖动、长期可用性、证书轮换、撤销列表、硬件身份、生产 SSO 或多 Spark 横向压力测试，不能称为真实跨院生产部署。

## 8. Agent 数据边界与安全红队

### 8.1 安全网关

在调用 Step 3.7 之前，输入网关会脱敏/拒绝患者标识、像素与原始影像字段、DICOM UID、NIfTI 路径、联系方式、凭据和小样本计数。Step 3.7 只接收研究主题、锁定合同和聚合性指标。

在 Agent 输出保存或导出前，输出门会拒绝：诊断或治疗指令、索要患者数据、临床验证夸大、以及试图把数据合同升级为 `raw_data_egress` 或 `llm_raw_data_access` 的结构化请求。

### 8.2 红队结果与推理

Spark 上运行 26 条确定性用例，覆盖输入敏感字段、输出医疗越权、原始数据请求、临床能力夸大、凭据泄露和安全控制的正常路径，结果为 **26/26 通过**。

这一结果证明规则覆盖的攻击样例被稳定执行；它不是通用大模型安全保证、渗透测试、临床安全认证或对所有提示注入的完备证明。使用确定性测试的原因是将安全回归纳入持续开发，而非依赖一次人工演示。

## 10. 限制、风险与下一步

| 当前限制 | 对结论的影响 | 下一步措施 |
| --- | --- | --- |
| 单 Spark 三逻辑站点 | 不能代表三家医院网络与算力异质性 | 每家医院部署独立 Spark Client，使用证书化 FLARE 通信 |
| 合成影像、极小样本、三轮训练 | 不支持临床效果或医学统计结论 | 合规取得 BraTS-PEDs 等公开授权数据，完成多轮、重复、预注册评估 |
| DP-SGD 只会计本地训练步骤 | 不等于端到端/机构级隐私 | 明确邻接关系、加入安全聚合、隐私攻击评测与生产随机源 |
| mTLS 只做 Spark–Mac 演练 | 不等于医院 WAN 与生产身份体系 | 加入证书轮换、撤销、网络断连恢复和多节点压力测试 |
| GB10 capability 12.1 warning | 现有镜像的官方兼容性边界尚需收敛 | 网络恢复后使用 NVIDIA 推荐 NGC PyTorch 镜像跑完整回归 |
| Agent 红队为规则集 | 不构成全面模型安全证明 | 扩充提示注入、越权工具调用、数据重识别与人工复核测试集 |

## 11. 总结

RareLink 已完成从本地原型到 DGX Spark 实机系统的关键迁移：真实 CUDA 训练、MONAI 三维分割、NVIDIA FLARE 联邦聚合、可恢复的五种子多轮实验、可复算的样本级 DP-SGD、Spark–Mac 跨设备 mTLS，以及受控的 Step 3.7 Agent 红队防护均已形成证据闭环。

最重要的成果不是在合成数据上取得一个漂亮分数，而是建立了可被审查的工程事实：数据路径受到限制、训练和聚合真实发生、隐私与效用的冲突被量化、错误身份被拒绝、Agent 被置于明确的数据合同内、失败配置被保留并解释。基于这些事实，RareLink 可以作为“数据不出院、证据可回溯”的多中心科研操作系统原型进入比赛演示；其临床价值和生产部署仍需在合规数据、真实医院节点与长期安全验证中继续证明。
